Make a Kubernetes cluster production-ready on Hetzner cloud using the Cluster API

  1. Back-End Development
Posted 8 days ago
6 required Connects (14 available)
Hetzner cloud production-ready cluster

We are looking for offers from people who know both Kubernetes and Go (Golang) very well. Knowledge of Linux and networking is also required.We want to build a highly available Kubernetes cluster on Hetzner cloud servers. For this we need the Cluster API (https://github.com/kubernetes-sigs/cluster-api). There are several examples for the Cluster API in action, e.g. for AWS (https://github.com/kubernetes-sigs/cluster-api-provider-aws), Azure (https://github.com/kubernetes-sigs/cluster-api-provider-azure) and Packet (https://github.com/packethost/cluster-api-provider-packet). They are all build using the very well documented framework of the Cluster API (https://cluster-api.sigs.k8s.io/).Much of your work has already been done (https://github.com/cornelius-keller/cluster-api-provider-hetznercloud). Basically, the existing provider integration followed 3.6.3 and 3.6.4 of the Cluster API book (https://cluster-api.sigs.k8s.io/).

What we need, is to make the Cluster API for Hetzner cloud servers production ready, as well as other features which allow for a production-ready Kubernetes cluster.

From you, we expect to know Kubernetes and Go very well. We can help you out on several of the below-mentioned tasks, but it would also be great if you are able to do all of them.

In the following, we describe our project plan:.

private Network

implementation of the web hooks in the provider integration

HA Floating IP for the control planes IPv4/IPv6, but Kubernetes without Dual Stack because private network is only IPv4.

Custom load balancer. We are using Floating IPs, MetalLB, Calico and can use this controller as basis for assigning the IPs to the nodes: https://github.com/cbeneke/hcloud-fip-controller. Here we have to implement to version of the controller one for the kubeapiserver and one for the load balancing with MetalLB, the challenge is to follow MetalLB election process.

IPv6 to IPv4 translation for all Floating IPs; gobetween for example

Bastion Host in HA mode https://docs.aws.amazon.com/quickstart/latest/linux-bastion/architecture.html

Implementation of the upcoming version v0.3.x (June-July). E2E test plan, Machine Health Checks, auto-scaler, etc. https://cluster-api.sigs.k8s.io/roadmap.html

OS configurations: Firewall, SELinux, Appamor, CRI and other security precautions; you will work together with our team on this feature. We follow the CIS Guidelines. The provider integration can not follow the best practices of the Cluster API with the use of images, because the provider Hetzner does not allow the use of own images. All configuration is made by scripts https://github.com/cornelius-keller/cluster-api-provider-hetznercloud/blob/master/controllers/templates.go and the use of PreKubeAdm of the CABPK (https://github.com/kubernetes-sigs/cluster-api/tree/master/bootstrap/kubeadm). Furthermore we have to implement here the things we defined on the Provider integration and in the template, like the private network and therefore the deactivating of the external IP address. On several points we have to adjust both the provider integration and the OS Configuration. To cover several templates and machineDeployments we have to find first the best structure to modularize and reuse the scripts or making use of PreKubeAdm and PostKubeAdm.

Cluster API Security; Management Cluster access control and preventing of deleting nodes accidentally, backups of etcd

Kubernetes Security and Cluster API Provider Development. We want to make the use of E2E Test for testing out the Provider Integration, the Kubernetes Conformance, the Cluster Security (several Test suites like CIS Kubernetes Benchmark).

Cluster API upgrades (Provider related, K8s Version, OS configuration) We think the best approach is to versionize with the Kubernetes Version and to separate all configuration of the Kubernetes and OS related config in directories. This way we have backward compatibility and can develop an upgrade path.

Template for creating control planes in all three data center, and one for creating control planes in only one data center. For testing reason to find out if we need Federation for multi cloud.1.For your information: We will also make use of the CCM https://github.com/hetznercloud/hcloud-cloud-controller-manager and CSI https://github.com/hetznercloud/csi-driver. The latter should be consumed by Rook.Useful links, just for your information:Cluster API:https://cluster-api.sigs.k8s.io/https://github.com/kubernetes-sigs/cluster-apiHetzner API:https://robot.your-server.de/doc/webservice/en.html#prefaceHetzner Floating IP Controller:https://github.com/cbeneke/hcloud-fip-controllerHetzner Cloud Controller Manager:https://github.com/hetznercloud/hcloud-cloud-controller-managerTools to build the API:https://book.kubebuilder.io/https://github.com/kubernetes-sigs/cluster-api-provider-docker/Image builder:https://image-builder.sigs.k8s.io/https://wiki.hetzner.de/index.php/Installimage/en
